两种 agent 都是实时写入 JSONL — 入库时机是核心设计决策

A

按需导入（yeek import / yeek sync）

用户手动或定期运行命令，扫描 agent 目录，增量导入新/变更的 session。

优势

  • 最简单可靠，用户可控
  • 不需要后台进程
  • 增量导入：只处理上次 sync 后新增的 session

劣势

  • 数据有延迟，不是实时的
  • 需要用户记得同步
B

文件监听（File Watcher 守护进程）

用 notify crate 监听 agent 目录变更，实时增量解析新写入的 JSONL 行。

优势

  • 数据始终最新
  • 可在 TUI 中实时显示活跃 session
  • 自动检测 session 开始和结束

劣势

  • 需要后台守护进程
  • 实时解析 JSONL 增加复杂度
  • 资源占用（虽然很小）
C

混合模式（推荐）

按需导入为基础 + 可选的文件监听增强。CLI 命令做全量/增量同步；TUI 模式下可选启动 file watcher 实时更新。两个模式共享同一套增量解析逻辑。

优势

  • 灵活：CLI 用户不需要守护进程
  • TUI 用户可享受实时更新
  • 增量解析逻辑复用
  • 渐进式实现：先做按需，再加 watcher

劣势

  • 两种模式需要保持一致

增量导入策略

无论选哪种模式，核心的增量导入逻辑相同：

  1. 发现 — 扫描 agent session 目录，与 yeek DB 中的记录对比
  2. 判断变更 — 比较文件 mtime 和已记录的 offset（已解析到第 N 行）
  3. 增量解析 — 只读取新增的 JSONL 行，更新 DB 中的 session 数据
  4. 状态标记 — 通过 agent 的活跃检测机制判断 session 是否仍在进行
